---
title: "flint start"
description: "Start the Flint daemon."
---

`flint start` launches the daemon process. The daemon manages all VM lifecycle — creating, killing, pausing, and resuming sandboxes — and exposes a REST API on `localhost:9100`.

You need the daemon running before you can use the Python SDK or any other `flint` commands.

## Usage

```bash
flint start [OPTIONS]
```

## Options

| Option | Default | Description |
|--------|---------|-------------|
| `--backend` | host default | Backend plugin (`firecracker`, `cloud-hypervisor`, `macos-vz`) |
| `--port` | `9100` | Port to listen on |
| `--data-dir` | `/microvms` (Linux), `~/Library/Application Support/flint/data` (macOS) | Directory for VM data (rootfs images, snapshots) |
| `--state-dir` | `/tmp/flint` (Linux), `~/Library/Application Support/flint/state` (macOS) | Directory for daemon state (db, pid file, logs) |

These can also be set via environment variables: `FLINT_BACKEND`, `FLINT_PORT`, `FLINT_DATA_DIR`, `FLINT_STATE_DIR`.

## Choosing a backend

Without `--backend` or `FLINT_BACKEND`, Flint picks a default based on your host: Firecracker on Linux, Virtualization.framework on macOS. If you have multiple backends installed, pick one explicitly:

```bash
sudo flint start --backend cloud-hypervisor
# or
FLINT_BACKEND=cloud-hypervisor sudo flint start
```

If you pass a backend name that isn't installed, `flint start` exits with code 2 and lists the available backends. Use [`flint backends list`](/cli/backends) to see what's ready on your host.

## Example

```bash
sudo flint start
```

On startup you'll see the daemon initialize the rootfs pool and build (or load) the golden snapshot. Once it prints the ready marker for your backend, the daemon is ready to accept requests.

## Running as a background process

```bash
sudo flint start &
```

The daemon will write its PID to `$FLINT_STATE_DIR/flintd.pid` so you can find it later.

## Notes

- On Linux, the daemon needs to run as root (or with `CAP_NET_ADMIN` and access to `/dev/kvm`) because it creates network namespaces and TAP devices.
- On macOS, the daemon re-execs itself under a Python binary signed with the Virtualization entitlement on first run.
- If the daemon crashes and restarts, it will attempt to recover any sandboxes that were running at the time using its SQLite state store. Each backend implements its own reattach logic.
